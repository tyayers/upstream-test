1. Create a test

Create a test in Upstream Test, and keep the generated id for future use.

Define what "healthy" means—from a simple 200 OK to a specific JSON body—and let UpstreamTest handle the rest. 


curl -X POST https://upstr.dev/tests \
  -H "Content-Type: application/yaml" \
  --data-binary @- << EOF

tests:
  - name: test host name
    path: /get
    verb: get
    assertions:
      - $.headers.Host===httpbin.org
EOF

Validate API Contracts

Ensure upstream APIs haven't introduced breaking changes. Define a JSON schema for the expected response and UpstreamTest will alert you if a field is missing, a data type changes, or a new field is added.

Catch errors before they break your application in production. 

# test/user-contract.yml
service: "user-service"
endpoint: "/users/123"

schema:
  type: "object"
  properties:
    id: { type: "number" }
    email: { type: "string", format: "email" }
    createdAt: { type: "string", format: "date-time" }
  required: ["id", "email"]

> upstream test --contract test/user-contract.yml
✖ user-service... FAIL
  - Response failed schema validation:
    - Missing required property: 'email'

Integrate with your CI/CD

Block bad deployments before they happen. Run upstream test as a required step in your GitHub Actions, GitLab CI, or Jenkins pipeline.

The CLI is designed to exit with a non-zero code on failure, halting any pipeline and preventing integration issues from shipping. 

# .github/workflows/ci.yml
name: "CI Pipeline"
on: [push]

jobs:
  integration-test:
    runs-on: "ubuntu-latest"
    steps:
      - name: "Checkout code"
        uses: "actions/checkout@v4"

      - name: "Install UpstreamTest"
        run: npm install -g upstream-test

      - name: "Run Upstream Tests"
        run: |
          # Run all tests in the ./upstream-tests/ folder
          upstream test ./upstream-tests/
